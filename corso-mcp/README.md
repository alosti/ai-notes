# Corso MCP

Materiale di un corso sul Model Context Protocol, il protocollo aperto di Anthropic per connettere LLM a strumenti e dati esterni. Il corso originale prevedeva moduli aggiuntivi sull'implementazione concreta di client e server MCP e sull'integrazione con l'host, sia dal punto di vista teorico che pratico. Qui sono presenti solo i primi due moduli: l'obiettivo era avere una comprensione sufficiente a valutare quando usare MCP, quando non usarlo e perché — non costruire un'implementazione. I due file coprono argomenti distinti — architettura e fondamentali nel primo, approfondimento sui tre primitivi nel secondo — ma ciascuno è leggibile indipendentemente dall'altro. Se sai già cos'è MCP e vuoi capire come funzionano tools, resources e prompts a livello operativo, il secondo file è più utile. Se parti da zero o vuoi il quadro completo del protocollo, il primo è il punto d'ingresso naturale.

## Contenuto

**01_mcp_fondamentali_architettura.md** — Perché esiste MCP e come è fatto

Il file parte dal problema che MCP risolve: prima del protocollo, integrare un LLM con strumenti esterni richiedeva reimplementazioni separate per ogni provider e ogni framework — N modelli × M strumenti = N×M integrazioni da mantenere. L'analogia usata è quella con LSP, il protocollo che ha standardizzato il supporto ai linguaggi negli editor. L'architettura è descritta nei suoi tre ruoli distinti — Host, Client, Server — con attenzione particolare a chi controlla cosa: il modello non parla mai direttamente con il server, l'host fa da mediatore e può bloccare o loggare qualsiasi chiamata. Il flusso a runtime è scomposto in cinque fasi, dalla discovery dei tool disponibili alla risposta finale all'utente, con codice Python e diagrammi che mostrano come l'LLM "veda" i tool MCP come se fossero function calling standard. Il protocollo base è JSON-RPC 2.0 — il file spiega perché non REST, con le ragioni concrete legate alla bidirezionalità e al paradigma action-based. I tre transport layer (stdio, SSE, Streamable HTTP) sono confrontati con scenari d'uso reali, incluse raccomandazioni specifiche per deployment su server OVH. La sezione finale confronta MCP con function calling puro, LangChain Tools e OpenAPI, con una valutazione onesta dei casi in cui MCP è la scelta giusta e di quelli in cui è overkill.

**02_mcp_tre_primitivi.md** — Il vocabolario del protocollo

Il file entra nel dettaglio dei tre primitivi — Tools, Resources, Prompts — con un livello di approfondimento che va oltre la panoramica del primo modulo. Per i tool, l'anatomia completa (name, description, inputSchema), il ciclo di vita di una tool call con i messaggi JSON-RPC corrispondenti, la distinzione tra errori di protocollo e errori applicativi (isError: true), e una sezione dedicata a come scrivere le description in modo che il modello capisca quando usare un tool e quando no — il campo più critico e quello su cui la maggior parte delle implementazioni sbaglia. Per le resource, gli URI scheme, la differenza tra resource statiche e template dinamici (RFC 6570), il meccanismo di subscribe e notifiche, e i quattro pattern di caricamento da parte dell'host: statico incondizionato, routing deterministico su keyword, routing semantico tramite LLM leggero, e resource prodotte come output di un tool. Per i prompt, l'anatomia della chiamata prompts/get, le condizioni in cui ha senso usarli rispetto a un system prompt hardcoded, e gli usi meno ovvi in architetture multi-step. Il file chiude con una mappa decisionale concreta — chi controlla l'attivazione del primitivo, con scenari reali su dominio finanziario — e con un'analisi dell'errore più comune: mettere tutto come tool, con i costi reali in termini di context window, latenza e affidabilità del modello.
